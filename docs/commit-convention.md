# 커밋 및 PR 컨벤션 (Commit & PR Convention)

이 문서는 코드 변경 이력을 명확히 남기고, 이슈 트래커(GitHub Projects)와의 자동화를 위한 커밋 규칙입니다.

## 1. 커밋 메시지 규칙
모든 커밋 메시지에는 관련된 이슈 번호를 대괄호 안에 포함해야 합니다. 단순 오타 수정이라도 해당 이슈의 문맥 안에 있다면 번호를 기입합니다.

> **Format: `[#이슈번호] 타입: 설명`**
> * Example 1: `[#62] feat: React + TS 프로젝트 초기 세팅`
> * Example 2: `[#59] refactor: MemberController RestController로 전환`
> * Example 3: `[#58] t/s: Repository 강결합 해제를 위한 EventListener 도입 시도`

## 2. PR (Pull Request) 생성 및 연동 규칙
코드 작성이 끝나고 `develop` 브랜치로 병합하기 위해 PR을 생성할 때, PR 본문(Description)에 반드시 아래 키워드를 포함해야 합니다.

* **Format:** `Closes #이슈번호`
* **작동 원리:** PR이 리뷰를 거쳐 병합(Merge)되는 순간, 연결된 이슈가 자동으로 `Closed` 처리되며 GitHub Projects 보드에서도 `Done`으로 이동합니다.
* **파생 이슈 발생 시:** "회원가입 전환 중 500 에러 발생함. 관련된 작업: #59" 처럼 본문에 `#원본이슈번호`를 언급하여 관계망을 형성합니다.

## 3. 컨벤션 수립의 목적과 필요성
1. **이력 추적 및 맥락 보존:** 나중에 "이 코드를 왜 짰지?" 궁금할 때 커밋의 이슈 번호를 누르면 과거의 고민(댓글)과 기획 의도로 즉시 이동하여 유지보수 비용 절약
2. **자동화:** `Closes #` 규칙을 통해 이슈 닫기, 보드 이동 관련 리소스 절약
