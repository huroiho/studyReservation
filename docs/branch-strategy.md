# 브랜치 전략 가이드 (Branch Strategy)

이 문서는 프로젝트의 안정적인 개발과 각 이슈의 추적을 위한 브랜치 운영 규칙입니다.

## 1. 브랜치 사이클 (Branch Flow)
우리 프로젝트는 아래의 핵심 브랜치를 기준으로 운영됩니다.
* **`main`**: 실제 서비스(Production) 환경에 배포되는 안정된 버전
* **`develop`**: 다음 배포를 위해 기능이 통합되고 테스트되는 기준 브랜치
* **`작업 브랜치`**: `develop`에서 분기하여 각 이슈를 처리하는 브랜치

## 2. 브랜치 네이밍 규칙 (Naming Convention)
이슈 번호를 포함하여 어떤 이슈를 처리하기 위한 브랜치인지 명확히 추적할 수 있게 합니다.

> **Format: `타입/도메인/이슈번호-설명`**
> * Example (기능 추가): `feat/member/12-jwt-filter`
> * Example (리팩토링): `refactor/payment/57-strategy-pattern`
> * Example (트러블슈팅): `t/s/config/54-cors-error`

### 타입 (Type) 기준 - 라벨과 동일
* **feat**: 새로운 기능 개발
* **refactor**: 기존 코드 구조 개선 (CSR 전환, 결합도 분리 등)
* **fix**: 버그 및 오류 수정
* **docs**: 문서 작성 (Swagger 명세, README 등)
* **t/s**: 트러블슈팅 및 환경 설정 해결 (bottleneck, config, domain)

## 3. 컨벤션 수립의 목적과 필요성
1. **가독성 확보:** 브랜치명에 긴 설명 대신 간결한 키워드와 이슈 번호를 넣어 가독성을 높였습니다.
2. **추적성 유지:** `브랜치명(57)` → `이슈(#57)` → `칸반 보드(진행 상황)` 흐름으로 연결되어, 커밋 내역만으로도 현재 작업의 명세와 진행도를 쉽게 파악할 수 있습니다.
